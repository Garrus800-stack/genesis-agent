# Genesis v6.0.7 — Earned Autonomy

## Core Finding

`TrustLevelSystem.checkApproval()` exists with a clean API but is **never called** from
`AgentLoop._requestApproval()`. The trust system was built, tested, documented — and then
never plugged into the actual decision gate. Every approval still goes to the user, regardless
of trust level.

The feedback loop from "I completed tasks successfully" to "I can skip asking" is broken.

## Novel Solution: Wilson-Score Earned Autonomy

Instead of coarse 4 trust levels, Genesis earns autonomy **per action type** using
Wilson score confidence intervals — the same algorithm Reddit uses for ranking.

**Why Wilson score?** Because it handles small samples correctly.
A naive success rate of 5/5 = 100% would auto-promote after 5 tries.
Wilson lower bound of 5/5 at 95% confidence = 0.57 — not enough.
Wilson lower bound of 45/50 at 95% confidence = 0.80 — earned.

This means Genesis can't game its own trust system with a lucky streak.
It has to *consistently* succeed before gaining autonomy.

## Changes (5 files modified, 1 new)

### 1. NEW: `EarnedAutonomy.js` (~180 LOC)
Phase 11 service. Per-action-type Wilson score tracker.

- Listens to `agent-loop:step-complete` events
- Records outcome per action type in a sliding window (200 outcomes max)
- Computes Wilson score lower bound (z=1.96, 95% confidence)
- When `wilson_lower > 0.85` AND `samples >= 30` → auto-promote
- Promotion = `TrustLevelSystem.acceptUpgrade(actionType)` 
- Emits `autonomy:earned`, `autonomy:revoked`
- Auto-revokes if wilson_lower drops below 0.70 (performance degradation)
- CLI: `/autonomy` — shows per-action confidence scores
- Persists via StorageService

### 2. MODIFY: `AgentLoop._requestApproval()` (~15 LOC change)
The actual wiring.

- Late-bind `trustLevelSystem` into AgentLoop
- In `_requestApproval(action, description)`:
  - Call `trustLevelSystem.checkApproval(action)`  
  - If `approved: true` → resolve immediately, emit `agent-loop:auto-approved`
  - If not → fall through to existing user approval flow
- Net effect: Once an action type earns trust, the user stops being asked

### 3. MODIFY: `OnlineLearner._checkStreak()` → Reactive Prescription
Currently: streak detected → emit event → nobody acts.
After: streak detected → emit `online-learning:adaptation-needed` → AdaptiveStrategy listens.

- AdaptiveStrategy gets new late-binding: `onlineLearner` (optional)
- AdaptiveStrategy.start() subscribes to `online-learning:adaptation-needed`
- On event: immediate `runCycle()` (bypasses IdleMind schedule)
- Gap closure: failure pattern → adaptation in seconds, not hours

### 4. MODIFY: `AutonomousDaemon` — Trust-Gated `autoOptimize`
One-line change with massive impact.

- In `_runCycle()`, check trust level before optimization:
  `const canOptimize = this._trustLevel >= 2;`
- At AUTONOMOUS (level 2+), the Daemon applies its own findings
- Still gated by SelfModificationPipeline's safety checks (AST scan, snapshot, verify)

### 5. MODIFY: `AgentLoopSteps._executeStep()` — Per-Step Trust Check
Before executing risky steps (SHELL, WRITE_FILE, SELF_MODIFY), check trust.

- Get step type → map to FormalPlanner action type
- Call `trustLevelSystem.checkApproval(actionType)`
- Auto-approved → execute silently
- Not approved → emit `agent-loop:approval-needed` (existing flow)
- This enables granular earned autonomy per step type

### 6. INFRA: Events + Schemas + Manifest
- EventTypes: +3 events (`autonomy:earned`, `autonomy:revoked`, `autonomy:status`)
- EventPayloadSchemas: +3 schemas + schemas for existing trust events
- Phase 11 manifest: EarnedAutonomy service registration
- Phase 8 lateBindings: trustLevelSystem → AgentLoop
- Phase 9 lateBindings: adaptiveStrategy → OnlineLearner (reverse)

## Testing Strategy

- EarnedAutonomy: Wilson score math, promotion threshold, revocation, persistence
- AgentLoop trust integration: auto-approve bypass, fallthrough to user
- OnlineLearner→AdaptiveStrategy bridge: streak → immediate cycle
- AutonomousDaemon trust gate: level 0/1 = no optimize, level 2+ = optimize

## Safety Invariants

1. Trust level 0 (SUPERVISED) always asks the user — no bypass possible
2. DEPLOY, EXTERNAL_API, EMAIL_SEND always need explicit approval below level 3
3. SelfModificationPipeline safety checks are INDEPENDENT of trust — always run
4. Wilson score auto-revocation prevents trust from persisting after degradation
5. All trust decisions are audit-logged with timestamp and reasoning
