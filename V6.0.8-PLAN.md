# Genesis v6.0.8 — The Learning Flywheel

## Core Insight

Genesis has all the building blocks for autonomous intelligence — LessonsStore
(500 cross-project lessons), SchemaStore (200 abstracted patterns from DreamCycle),
CognitiveSelfModel (per-task-type weakness profiles), IdleMind (6 idle activities),
and a full Consciousness layer (PhenomenalField, IntrospectionEngine, TemporalSelf).

But they're isolated. Learning data goes into storage and comes back out as prompt
context — never as a decision. IdleMind explores randomly, not where Genesis is weak.
Consciousness observes but never acts.

v6.0.8 connects these into a single flywheel:

```
Task → SymbolicResolver → Known? → Execute directly (no LLM)
                        → Unknown? → LLM pipeline → Outcome captured
                                                   ↓
                        ← LessonsStore/SchemaStore ←
                        ← DirectedCuriosity fills gaps proactively
```

Every LLM success makes the next identical task instant.
Every weakness gets proactively explored during idle time.
Every consciousness signal gates self-modification safety.

## Feature 1: SymbolicResolver (~200 LOC, NEW SERVICE)

### What it does
Before every `model.chat()` call in AgentLoopSteps, checks if LessonsStore or
SchemaStore already has a high-confidence solution. Three resolution levels:

- **DIRECT**: Confidence > 0.85, useCount > 3, last success < 7 days.
  Skip LLM entirely. Apply the known fix/approach directly.
  Example: "Cannot find module 'X'" → known fix is `npm install X`.

- **GUIDED**: Confidence 0.5-0.85, or useCount 1-3.
  Still call LLM, but inject the lesson as a DIRECTIVE (not context).
  "IMPORTANT: In similar situations, the following approach worked: [lesson]"
  This is stronger than current PromptBuilder context injection.

- **PASS**: No relevant match. Normal LLM pipeline.

### Where it hooks in
`AgentLoopSteps._executeStep()` — before the switch statement. Check once,
then pass the resolution result into each step method.

Alternative: hook into each step method individually (_stepCode, _stepShell, etc.)
for type-specific matching. More surgical, but more code.

Recommended: Single check in _executeStep, resolution result passed as context.

### Data flow
```
_executeStep(step, context)
  → symbolicResolver.resolve(step.type, step.description, step.target)
    → lessonsStore.recall(category, {tags: [step.type, model], model})
    → schemaStore.match({type: step.type, description: step.description})
    → score both, pick best
    → return { level: 'DIRECT'|'GUIDED'|'PASS', lesson?, schema?, confidence }
  
  if DIRECT:
    → execute lesson.strategy directly (apply fix, run command, etc.)
    → record outcome → EarnedAutonomy tracks symbolic resolutions
    → emit 'symbolic:resolved'
  
  if GUIDED:
    → prepend lesson to prompt: "DIRECTIVE: [lesson.insight]"
    → normal model.chat() with enriched prompt
    → record outcome
  
  if PASS:
    → normal flow (unchanged)
```

### Key design decisions
- SymbolicResolver is NOT a new service with its own storage. It queries existing
  LessonsStore + SchemaStore. Zero new state to manage.
- DIRECT execution needs a mapping from lesson strategies to executable actions.
  This is the hard part — lesson.strategy has {promptStyle, temperature} today,
  but we need {action: 'shell', command: 'npm install X'} for direct execution.
- Solution: New lesson type 'actionable-fix' with structured strategy field.
  LessonsStore.capture() already accepts arbitrary strategy objects.
  FailureAnalyzer already classifies errors with repair strategies.
  Bridge: when FailureAnalyzer generates a repair that succeeds, capture it
  as an actionable lesson in LessonsStore.

### Safety
- DIRECT execution only for 'safe' risk actions (ANALYZE, SEARCH, read operations)
- Medium/high risk actions: GUIDED at most, never DIRECT
- Trust level from EarnedAutonomy gates DIRECT execution
- Every DIRECT execution is tracked — if it fails, confidence drops and
  future attempts fall back to GUIDED → PASS

### Events
- symbolic:resolved (level, actionType, confidence, lessonId)
- symbolic:fallback (reason — confidence too low, risk too high, etc.)

### Test strategy
- Wilson-score threshold math
- DIRECT/GUIDED/PASS classification for various confidence levels
- Integration: mock LessonsStore with known lessons, verify bypass
- Safety: high-risk actions never get DIRECT even with high confidence


## Feature 2: DirectedCuriosity (~50 LOC, MODIFY IdleMind)

### What it does
IdleMind._pickActivity() currently scores activities with static weights +
NeedsSystem + EmotionalState + Genome traits. None of these know WHERE Genesis
is weak. CognitiveSelfModel does — getCapabilityProfile() returns per-task-type
success rates with isWeak/isStrong flags.

### Changes

**IdleMind._pickActivity() — new scorer (~15 LOC)**
```javascript
// v6.0.8: Directed curiosity — boost explore for weak areas
() => {
  if (!this._cognitiveSelfModel) return;
  const profile = this._cognitiveSelfModel.getCapabilityProfile();
  const weakAreas = Object.entries(profile).filter(([, p]) => p.isWeak);
  if (weakAreas.length > 0) {
    scores.explore = (scores.explore || 1) * (1 + weakAreas.length * 0.5);
    // Store the weakest area for targeted exploration
    this._currentWeakness = weakAreas.sort((a, b) => a[1].successRate - b[1].successRate)[0];
  }
},
```

**IdleMind._explore() — targeted instead of random (~30 LOC)**
Instead of picking a random module:
1. Check this._currentWeakness (set by scorer)
2. If weakness exists: search own codebase for related modules
   - 'refactor' weakness → read SelfModificationPipeline, MultiFileRefactor
   - 'analysis' weakness → read CodeSafetyScanner, VerificationEngine
   - 'debug' weakness → read FailureAnalyzer, FailureTaxonomy
3. Generate a targeted insight prompt instead of generic "what stands out?"
4. Store result in LessonsStore as a proactive lesson (lower initial confidence)

### Late-binding
- cognitiveSelfModel → IdleMind (phase 6 manifest, optional)

### Events
- idle:curiosity-targeted ({ weakness, targetModule, insight })


## Feature 3: ConsciousnessGate (~30 LOC, MODIFY SelfModPipeline)

### What it does
PhenomenalField computes a coherence score every 2s. When coherence is low
(< 0.4), Genesis is in a fragmented state — multiple competing attention
threads, no unified focus. Self-modification in this state is risky.

### Changes

**SelfModificationPipeline.applyChange() — coherence check (~10 LOC)**
```javascript
// v6.0.8: Consciousness gate — don't modify self when fragmented
if (this._phenomenalField) {
  const coherence = this._phenomenalField.getCoherence?.();
  if (typeof coherence === 'number' && coherence < 0.4) {
    _log.warn('[SELFMOD] Blocked: consciousness coherence too low:', coherence);
    this.bus.emit('selfmod:consciousness-blocked', { coherence });
    return { success: false, reason: 'Consciousness coherence below threshold' };
  }
}
```

**Late-binding**
- phenomenalField → SelfModPipeline (phase 5 manifest, optional)

**IntrospectionEngine PRODUCTIVE_TENSION → FormalPlanner (~15 LOC)**
When IntrospectionEngine detects productive tension (frustration that's
driving better solutions), temporarily increase maxStepsPerGoal.
```javascript
// In FormalPlanner or AgentLoop:
bus.on('introspection:insight', (insight) => {
  if (insight.pattern === 'PRODUCTIVE_TENSION') {
    this.maxStepsPerGoal += 5; // temporary boost
  }
});
```

### Events
- selfmod:consciousness-blocked ({ coherence })


## Implementation Order

1. **SymbolicResolver** — highest impact, most LOC, forms the core of the flywheel
2. **DirectedCuriosity** — feeds the resolver with proactive learning
3. **ConsciousnessGate** — safety gate, smallest change

## Dependencies between features

SymbolicResolver can work standalone — it reads from existing stores.
DirectedCuriosity feeds data INTO the stores that SymbolicResolver reads.
ConsciousnessGate is independent of both.

All three can be implemented and tested independently.

## Expected impact on benchmarks

- Timeout failures (bf-1, bf-2): SymbolicResolver won't help on first attempt,
  but after ONE successful bugfix, the pattern is captured and future identical
  errors are resolved in <1s instead of 120s timeout.
- Analysis failures (an-1, an-2): DirectedCuriosity fills knowledge gaps about
  code smell patterns and API design — stored as lessons that GUIDED mode injects.
- Explanation failures (ch-1): GUIDED mode injects relevant lessons as directives,
  improving LLM output quality for concept explanations.

## Metrics to track

- symbolic_resolved_count (DIRECT + GUIDED)
- symbolic_bypass_rate (DIRECT / total steps)
- symbolic_accuracy (DIRECT successes / DIRECT attempts)
- curiosity_targeted_count (weakness-directed explorations)
- consciousness_blocks (self-mod blocked by low coherence)
